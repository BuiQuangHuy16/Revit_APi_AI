## Project Overview

Autodesk Revit add-in targeting Revit versions 2023–2027. The same C# codebase compiles against multiple Revit API versions using solution configurations and conditional compilation directives.

## Build Commands

**IDE (JetBrains Rider or Visual Studio):**
- Select a solution configuration from the dropdown (e.g., `Debug.R27`, `Release.R25`)
- Build → Build Solution
- The `Debug` run button launches Revit with the add-in attached

**Command line (from repo root):**
```shell
# Compile only
cd RevitAPP_AI/build; dotnet run

# Compile + create MSI installer + Autodesk bundle
cd RevitAPP_AI/build; dotnet run -- pack
```

No automated test suite exists — testing is manual against a running Revit instance.

## Solution Structure

```
RevitAPP_AI/
  Application/     # Main add-in project (Aplication.csproj)
  build/           # ModularPipelines build automation
  install/         # WixSharp MSI installer generator
  source/          # Placeholder for additional projects
  output/          # Generated bundles and installers (git-ignored)

```

## Application Architecture

**Entry point:** `Application/Application.cs` — implements `IExternalApplication`, handles `OnStartup` (ribbon creation, Serilog logger init) and `OnShutdown` (log flush).

**Commands:** `Application/Commands/` — each file is an `IExternalCommand` registered on the ribbon in `Application.cs`. `StartupCommand.cs` is the placeholder command wired to the "Execute" ribbon button.

**Intended MVVM layout** (add folders as needed):
- `Commands/` — ribbon-invoked commands
- `Models/` — data/DTOs
- `ViewModels/` — bindable properties and commands
- `Views/` — WPF XAML views
- `Utils/` — shared helpers and extensions
- `Resources/Icons/` — 16px and 32px ribbon icons

## Multi-Version Targeting

Solution configurations follow the pattern `{Debug|Release}.R{YY}` where `YY` is the last two digits of the Revit year (e.g., `Release.R27` = Revit 2027).

The Revit MSBuild SDK (`Nice3point`) automatically sets `TargetFramework` from the `RevitVersion` extracted from the configuration name:
- Revit 2023–2024 → `.NET Framework 4.8`
- Revit 2025–2026 → `.NET 8.0`
- Revit 2027 → `.NET 10.0`

**Conditional compilation constants** (generated by the SDK):
```csharp
#if REVIT2027
    // Revit 2027 only
#endif

#if REVIT2025_OR_GREATER
    // Revit 2025, 2026, 2027 ...
#endif

#if !REVIT2023_OR_GREATER
    // Revit 2022 and below
#endif
```

**NuGet version wildcards** — always use `Version="$(RevitVersion).*"` for Revit API packages so the correct version resolves automatically per configuration:
```xml
<PackageReference Include="Nice3point.Revit.Api.RevitAPI" Version="$(RevitVersion).*"/>
```

## Managing Supported Versions

To add or remove a Revit version:
1. Edit `.csproj` **manually** (IDEs can corrupt configurations) — add/remove entries in the `<Configurations>` property.
2. Edit the `.slnx` solution file — add/remove the corresponding `Debug.RXX` and `Release.RXX` configuration entries.

## Key Dependencies

- `Nice3point.Revit.Toolkit` / `Nice3point.Revit.Extensions` — Revit API helpers
- `Serilog` with Debug sink — logging
- `ModularPipelines.DotNet` — build automation (in `build/`)
- `WixSharp` — MSI installer generation (in `install/`)
- `ILRepack` — merges assemblies into a single DLL for distribution

## Build Pipeline (ModularPipelines)

`build/Modules/` contains pipeline stages run in order:
1. `ResolveVersioningModule` — reads version from GitVersion
2. `ResolveConfigurationsModule` — determines active Revit version targets
3. `CleanProjectModule` — cleans previous outputs
4. `CompileProjectModule` — compiles for all configurations
5. `CreateBundleModule` — creates Autodesk `.bundle` package
6. `CreateInstallerModule` — produces `*-SingleUser.msi` and `*-MultiUser.msi`

Versioning follows semantic versioning via `GitVersion.yml` (ContinuousDeployment strategy).
